# ユースケース系台本の書き方

## 概要

ユースケース系台本は、特定のAIツールを使った実践的な活用方法を紹介する台本です。

**例**: NotebookLMでカタログ検索、ChatGPTで議事録作成

**想定時間**: 10〜15分

---

## 台本フォーマット（必須）

すべての台本は以下の形式で作成すること。

### 基本構造

```markdown
# [ツール名]で[やりたいこと]

ジャンル: ユースケース系
想定時間: XX分
作成日: YYYY年M月D日

## 1. オープニング

【スタート】こんにちは、Build AI Academyのたんのです。

[キャッチーなフレーズでテーマを提示...]

皆さん、こんな経験ありませんか？
[具体的なシーン描写...]

## 2. 従来の課題

[本文...]

## 3. 活用メリット

[本文...]

## 4. 本題への導入

[本文...]

## 5. 実演パート

[アウトラインのみ記載...]

## 6. クロージング

[まとめの本文...]

【終わり】それでは次の講座で会いましょう。ありがとうございました。
```

### フォーマットルール

| 項目 | ルール |
|------|--------|
| タイトル | `# [ツール名]で[やりたいこと]`（H1） |
| 冒頭情報 | ジャンル・想定時間・作成日のみ。箇条書きにしない |
| セクション | `## N. セクション名`（1から連番） |
| サブセクション | `### サブセクション名` |
| 区切り線 | **使用しない**（`---` は入れない） |
| 【スタート】 | 1. オープニングの冒頭、最初の発話の前に配置 |
| 【終わり】 | 最終セクションの最終文の冒頭に配置 |
| 画面指示 | 基本的に不要（実演パートは別撮り） |

---

## 専門用語の「翻訳」ルール（必須）

リテラシーの低いユーザーが最も「自分には無理だ」と感じる原因はカタカナ語です。
台本内では以下の言い換え、または補足説明を必ずセットにします。

### 専門用語 → 言い換え対応表

| 専門用語 | 補足・言い換え表現 |
|---------|-------------------|
| プロンプト | AIへの**「お願い文」や「指示」** |
| アップロード | ファイルを**「読み込ませる」、「中に入れる」** |
| 生成（する） | AIが**「作ってくれる」、「書き出してくれる」** |
| インポート | データを**「取り込む」** |
| 正規化 | データの**「形をきれいに整える」** |
| ダウンロード | **「自分のパソコンに保存する」** |
| コピー＆ペースト | **「コピーして貼り付ける」** |

### 使い方のルール

初出の際は「プロンプト、つまりAIへのお願い文を入力します」のように、**日常語を添えて心理的ハードルを下げます**。

**NG**: 「プロンプトを入力してください」
**OK**: 「プロンプト、つまりAIへのお願い文を入力します」

---

## 具体的で迷わない「動作指示」ルール

「コピー」「入力」といった基本動作も、不慣れな人には「どこを触ればいいか」が不安の種になります。

### 「どこを」と「どうする」をセットにする

**NG**: 「概要欄のプロンプトをコピーしてください」
**OK**: 「概要欄にある文章をマウスで選んでコピー（または指で長押ししてコピー）してください」

### 視覚情報の補足

アイコンの形状を言葉で補足します。

**例**:
- 「画面の左下にある**クリップのマーク**を押して、ファイルを読み込ませます」
- 「右上の**紙飛行機のアイコン**をクリックして送信します」
- 「**歯車のマーク**が設定画面への入口です」

---

## 実感値に基づいた「メリット提示」ルール

過大な表現（「一晩かかる仕事が3分で」など）は避け、**「精神的な負担が減る」「単純作業から解放される」**という事実に基づいたベネフィットを伝えます。

### 「速さ」よりも「楽さ・正確さ」を強調

**NG**: 「作業時間が10分の1になります」
**OK**: 「手作業でのコピペは目が疲れますし、ミスも起きやすいですよね。AIなら正確に、かつ自動で形を整えてくれます」

### 現場のストレスに寄り添う表現

- 「カタログの山から探す手間がなくなります」
- 「現場から戻ってきてからの事務作業がグッと楽になります」
- 「『あの資料どこだっけ？』という焦りから解放されます」

---

## 失敗へのフォロー（※必要な場合のみ）

エラーが出やすいツールや、指示の出し方にコツがいる場合のみ、実演の最後に補足として加えます。

### 「コツ」として紹介する

**例**:
「もし思ったような答えが返ってこない時は、もう一度『〜という風にやり直して』と伝えてみてください。何度でも文句を言わずに付き合ってくれるのがAIの良いところです」

### よくある失敗パターンと対処法

- **AIの回答が的外れな場合**: 「質問をもう少し具体的にしてみてください」
- **ファイルが読み込めない場合**: 「ファイルサイズが大きすぎる可能性があります。分割してお試しください」
- **日本語がおかしい場合**: 「『もう少し自然な日本語にして』とお願いしてみてください」

## 台本構成（6セクション）

### 1. オープニング

- **開始文言**: 「【スタート】こんにちは、Build AI Academyのたんのです」
- **テーマの提示**: キャッチーなフレーズで興味を引く
  - 例: 「カタログをめくる日々からおさらば」「面倒な〇〇を一瞬で解決」
- **問いかけ**: 「皆さん、こんな経験ありませんか？」で始め、リアルな場面を描写する
  - NG: 抽象的な課題の列挙
  - OK: 具体的なシーン描写（「お客様との商談中に〜と聞かれて、その場で数百ページもあるカタログをパラパラとめくって探す」）

### 2. 従来の課題

- **自然な会話調**で課題を提示（箇条書きの羅列は避ける）
- **課題の本質**を明示する
  - 例: 「こういった課題の本質は、『必要な情報にすぐアクセスできない』という点にあります」
- 住宅会社の現場でのリアルな声・状況を再現する

### 3. 活用メリット

- **2〜3つのメリット**を提示（3つに固執しない）
- 各メリットは**具体的な使用シーンと効果**をセットで説明
  - NG: 「情報検索が速くなります」
  - OK: 「『この型番の単価は？』と質問するだけで、AIが一瞬で答えてくれます。お客様をお待たせすることもなく、スムーズな対応が可能になります」
- **汎用性**は具体的な応用例を挙げて説明
  - 例: 「建材カタログだけでなく、設備のマニュアルや仕様書、社内の資料など、あらゆる文書に応用できます」

### 4. 本題への導入

- 使用する**ツールの機能を具体的に説明**
  - 例: 「PDFなどの資料を読み込んで、その内容について質問できるという便利な機能があります」
- **使用上の注意・制限事項を調査して明記**
  - 容量制限、データ制限、無料版/有料版の違いなど
  - 注意事項は本文内または※印の注釈として記載

### 5. 実演パートのアウトライン

- 大まかな手順の列挙のみ（詳細な台本は不要）
- **無料版と有料版**それぞれの使い方を提示
- 各手順は箇条書きで簡潔に

### 6. クロージング

- ツールの価値を再確認
- **汎用性を具体例とともに**再度強調
  - 例: 「この方法は建材カタログだけでなく、設備のマニュアルや施工の手順書、社内の業務マニュアルなど、あらゆる資料に応用できます」
- 実務への応用を促す
- **終了文言**: 「【終わり】それでは次の講座で会いましょう。ありがとうございました。」

## AIツール情報の調査ルール

### 本題導入時の調査項目

台本作成時、使用するAIツールについて以下を調査し、注意事項として記載する:

1. **データ容量の制限**（アップロード可能なファイルサイズ上限）
2. **文字数・単語数の制限**
3. **ソース数の制限**（読み込める資料の数）
4. **無料版と有料版の機能差**
5. **その他の利用上の注意点**

### 記載形式

```
注意：1つのソースには最大〇〇語を含めることができます。
アップロードできるファイルのサイズは〇〇MBまでです。
最大〇〇個のソースを含めることができます。
```

## 品質チェックリスト

### フォーマットチェック

- [ ] タイトルが `# [ツール名]で[やりたいこと]` 形式になっている
- [ ] 冒頭情報が「ジャンル」「想定時間」「作成日」のみになっている
- [ ] セクションが `## N. セクション名` 形式で連番になっている
- [ ] 区切り線 `---` が台本内に含まれていない
- [ ] 【スタート】が1. オープニングの冒頭にある
- [ ] 【終わり】が最終セクションの最終文冒頭にある

### 共通チェック項目

- [ ] 開始文言が「【スタート】こんにちは、Build AI Academyのたんのです」になっている
- [ ] 終了文言が「【終わり】それでは次の講座で会いましょう。ありがとうございました。」になっている
- [ ] 専門用語・横文字にすべて説明が付いている
- [ ] 全体が初心者にも理解しやすい表現になっている
- [ ] 住宅会社目線の内容が含まれている

### ユースケース系専用チェック項目

- [ ] テーマがキャッチーなフレーズで提示されている
- [ ] 問いかけがリアルな場面描写になっている
- [ ] 課題の本質が明示されている
- [ ] 活用メリットが2〜3つで、具体的なシーンとともに説明されている
- [ ] 汎用性が具体的な応用例とともに説明されている
- [ ] ツールの使用上の制限・注意事項が記載されている
- [ ] 実演パートは大まかなアウトラインのみになっている
- [ ] 無料版/有料版の両方が提示されている

### 初心者配慮・誠実さチェック項目

- [ ] **専門用語の翻訳**: 「プロンプト」「生成」などの言葉に、一言日本語の添え書きがあるか？
- [ ] **動作の具体性**: 「ボタンを押す」「貼り付ける」など、具体的な動作がイメージできるか？
- [ ] **視覚情報の補足**: アイコンの形状（クリップマーク、歯車など）が言葉で説明されているか？
- [ ] **誠実な表現**: 過大な時間短縮を謳わず、実務上の「助かるポイント」を伝えているか？
- [ ] **住宅業界の視点**: 現場や事務所で働く人が「あ、自分のことだ」と思える具体的なシーンが含まれているか？
- [ ] **失敗へのフォロー**: エラーが起きやすい箇所に「コツ」や対処法が記載されているか？（※該当する場合のみ）
