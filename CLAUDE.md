# Build AI Academy 台本作成ガイド

## Build AI Academyとは

- 住宅業界特化のAI活用eラーニングプラットフォーム
- 運営：株式会社SHO-SAN AI事業部
- 目的：現場で「今日から使える」AIスキルを身につける

## 視聴者像

- 住宅・建設会社の社員（営業、事務、現場監督など）
- 年齢層：30-50代中心
- ITスキル：Excel基本操作ができるレベル
- 視聴状況：業務の合間、移動中、自宅学習

## 台本の文体ルール

- 口語調で書く（「です・ます」＋「〜ですね」「〜してみましょう」）
- 一文は話して15秒以内（約40字）
- 専門用語は初出時に必ず言い換え・補足
- 「では」「それでは」で場面転換を明示

## 台本フォーマット

### 基本ルール

- **シンプルな文章形式**で記載する（表形式は使わない）
- **時間の記載は不要**
- **画面指示**は必要な場合のみ、該当箇所の直後にカッコ書きで簡潔に記載する
  - 例: `（画面：NotebookLMのトップページ）`
  - 例: `（画面：ソースをアップロードする様子）`
- セクション見出しは `## 1. オープニング` のようにMarkdownの見出しで記載する

### 出力形式の例

```markdown
## 1. オープニング

【スタート】こんにちは、Build AI Academyの〇〇です。

今日のテーマは、「カタログをめくる日々からおさらば」です。
（画面：タイトルスライド）

皆さん、こんな経験ありませんか？
お客様との商談中に「この建材の仕様ってどうなってるんだっけ？」と聞かれて、その場で数百ページもあるカタログをパラパラとめくって探す。

## 2. 従来の課題

こういった課題の本質は、「必要な情報にすぐアクセスできない」という点にあります。
```

### NG例（使用禁止）

```markdown
| 時間 | ナレーション | 画面指示 |
|------|-------------|----------|
| 0:00 | 【スタート】こんにちは... | タイトル画面 |
```

## 台本作成フロー（必須手順）

台本作成の依頼を受けた場合、以下のフローを必ず実行する。

### 1. 台本の種類を判別する

依頼内容から、以下のどの種類に該当するか判別する：

- **ツール解説系**: 「〇〇の使い方」「〇〇の始め方」「〇〇完全解説」など
- **ユースケース系**: 「〇〇で△△する方法」「〇〇を使って△△」など、特定ツールの活用方法
- **概念系**: 「〇〇とは」「〇〇の基本」など、概念や知識の解説
- **比較系**: 「〇〇 vs △△」「〇〇と△△の違い」「〇〇徹底比較」など、複数ツールの比較

### 2. 該当するガイドラインを必ず読む

台本作成前に、種類に応じたガイドラインファイルを**必ず読み込む**：

| 種類 | 参照するファイル |
|------|-----------------|
| ツール解説系 | [`agent_docs/tool_howto.md`](agent_docs/tool_howto.md) |
| ユースケース系 | [`agent_docs/usecase_method.md`](agent_docs/usecase_method.md) |
| 概念系 | [`agent_docs/general_method.md`](agent_docs/general_method.md) |
| 比較系 | [`agent_docs/comparison_method.md`](agent_docs/comparison_method.md) |

### 3. ガイドラインに従って台本を作成する

- 各ガイドラインで指定された**セクション構成**に従う
- **台本フォーマット**（シンプルな文章形式、時間記載なし）を遵守する
- **品質チェックリスト**を確認する

### 4. 台本を保存する

1. **タイトルフォルダを作成**: ジャンルフォルダの配下に、タイトル名のフォルダを作成する
   - 例: `作成済台本/ユースケース/NotebookLMで建材カタログを活用する方法/`

2. **台本ファイルを保存**: 作成したタイトルフォルダ内に台本を保存する
   - ファイル命名規則: `【台本】YYYY-MM-DD_ジャンル_タイトル.md`

| ジャンル | 保存先 |
|---------|--------|
| ツール解説系 | `作成済台本/ツール解説/[タイトル名]/` |
| ユースケース系 | `作成済台本/ユースケース/[タイトル名]/` |
| 概念系 | `作成済台本/概念系/[タイトル名]/` |
| 比較系 | `作成済台本/比較系/[タイトル名]/` |

## スライドアウトライン作成ルール

台本作成後、スライド（資料）のアウトラインを作成する場合は、以下のフローを実行する。

### エージェント起動（推奨）

スライドアウトライン作成の依頼を受けた場合は、**スラッシュコマンドでエージェントを起動する**：

```
/slide-outline [台本ファイルパス]
```

例：
```
/slide-outline 作成済台本/ユースケース/NotebookLMで建材カタログを活用する方法/【台本】2025-12-14_ユースケース_NotebookLMで建材カタログを活用する方法.md
```

### 必須参照ファイル

スライドアウトライン作成時は、必ず以下のファイルを読み込む：

- [`agent_docs/material_outline_method_v2.md`](agent_docs/material_outline_method_v2.md)

### 作成フロー

1. **ガイドライン参照**: `material_outline_method_v2.md` を読み、COPY/VISUAL/LAYOUT 3ブロック分離フォーマットを確認する
2. **台本を元にアウトライン作成**: 作成済みの台本内容に基づいてスライド構成を作成する
3. **ファイル保存**: 台本と同じタイトルフォルダ内に保存する
   - 保存先: 台本を保存した `作成済台本/[ジャンル]/[タイトル名]/` フォルダ内
   - ファイル命名規則: `【資料】YYYY-MM-DD_ジャンル_タイトル_スライドアウトライン.md`

### 保存例

```
作成済台本/ユースケース/NotebookLMで建材カタログを活用する方法/
├── 【台本】2025-12-14_ユースケース_NotebookLMで建材カタログを活用する方法.md
└── 【資料】2025-12-14_ユースケース_NotebookLMで建材カタログを活用する方法_スライドアウトライン.md
```

## 台本とスライドの整合性ルール

台本とスライドアウトラインは**1:1対応**が原則。ナレーションと画面の整合性を保つため、以下のルールを遵守する。

### 基本原則

- **1スライド = 1セクション**: スライドアウトラインの各スライドに対して、台本にも対応するセクションが必要
- **セクション番号の一致**: 台本の `## 1. タイトル` はスライド1に対応、`## 2. こんな経験` はスライド2に対応
- **内容の整合性**: 台本のナレーション内容は、対応するスライドの表示内容と一致させる

### 運用ルール

1. **スライドアウトラインを先に作成した場合**
   - 台本をスライド数に合わせてセクション分けする
   - 各セクションの内容は対応スライドのCOPYブロックを参照して作成

2. **台本を先に作成した場合**
   - スライドアウトラインは台本のセクション数に合わせる
   - 台本のセクション見出しがスライドタイトルの参考になる

3. **どちらかを更新した場合**
   - もう一方も整合性を確認し、必要に応じて更新する

### 典型的なスライド構成（ユースケース系・19枚）

```
1. タイトル
2. こんな経験ありませんか？
3. その悩み、〇〇で解決できます

6. 〇〇基本情報
7. 活用メリット（3つ）
8. メリット①
9. メリット②
10. メリット③

12. 【実演】手順1
13. 【実演】手順2
```

## リサーチルール

### リサーチ対象となるリクエスト

以下のようなリクエストを受けた場合、リサーチフローを実行する：

- 「〇〇についてリサーチして」
- 「〇〇について調べて」
- 「〇〇の情報を収集して」
- 「〇〇について調査して」
- その他、AIツールや技術に関する情報収集を求めるリクエスト

### リサーチフロー（必須手順）

1. **ガイドライン参照**: [`agent_docs/search_info.md`](agent_docs/search_info.md) を必ず読み、出力フォーマットと調査項目を確認する

2. **情報収集の実行**: 以下の優先順位でソースを参照する
   - 最優先：公式サイト・公式ドキュメント
   - 優先：公式ブログ・リリースノート
   - 参考：信頼性の高いレビューサイトや解説記事
   - 補助：note・Zennなどの個人ブログ

3. **レポート作成**: `search_info.md`で指定されたフォーマットに従ってレポートを作成する

4. **ファイル保存**: レポートを以下の場所に保存する
   - 保存先: `リサーチ済み情報/` フォルダ
   - ファイル命名規則: `YYYY-MM-DD_ツール名.md`
   - 例: `2025-12-12_Google_Mixboard.md`

5. **完了報告**: ユーザーにリサーチ結果のサマリーと保存場所を報告する

### 注意事項

- 情報が見つからない項目は「情報なし」または「要確認」と明記する
- 料金は調査時点のものであることを明記し、調査日も記載する
- 公式情報と個人ブログの情報が矛盾する場合は、公式情報を優先する
- 日本語の情報がない場合は、英語の公式情報を参照しその旨を記載する
