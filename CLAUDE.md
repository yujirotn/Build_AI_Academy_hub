# Build AI Academy 台本作成プロジェクト

## 概要

Build AI Academyは住宅業界向けのe-learningサービスです。
このリポジトリでは、コンテンツの台本を作成・管理します。

## フォルダ構成

- `作成済台本/`
  - `ユースケース/` - ユースケース系の台本を格納
  - `概念系/` - 概念系の台本を格納

---

## 台本作成ルール

### ファイル形式
- 台本はMarkdown（.md）ファイルで作成する

### 必須記載項目
- **作成日**: 台本が作成された日付（YYYY-MM-DD形式）
- **タイトル**: 台本のタイトル

### ファイル命名規則
- `YYYY-MM-DD_タイトル.md` の形式で保存する

---

## 台本の種類

台本は大きく2種類に分類される。種類によって構成・書き方が異なる。

| 種類 | 説明 | 例 |
|------|------|-----|
| **ユースケース系** | 特定のAIツールを使った実践的な活用方法を紹介 | NotebookLMでカタログ検索、ChatGPTで議事録作成 |
| **概念系** | AIの基礎知識や考え方、業界動向などを解説 | 生成AIとは何か、プロンプトの基本 |

---

## ユースケース系台本の構成（6セクション）

### 1. オープニング（約2分）

- **開始文言**: 「【スタート】こんにちは、Build AI Academyの〇〇です」
- **テーマの提示**: キャッチーなフレーズで興味を引く
  - 例: 「カタログをめくる日々からおさらば」「面倒な〇〇を一瞬で解決」
- **問いかけ**: 「皆さん、こんな経験ありませんか？」で始め、リアルな場面を描写する
  - NG: 抽象的な課題の列挙
  - OK: 具体的なシーン描写（「お客様との商談中に〜と聞かれて、その場で数百ページもあるカタログをパラパラとめくって探す」）

### 2. 従来の課題（約1分30秒）

- **自然な会話調**で課題を提示（箇条書きの羅列は避ける）
- **課題の本質**を明示する
  - 例: 「こういった課題の本質は、『必要な情報にすぐアクセスできない』という点にあります」
- 住宅会社の現場でのリアルな声・状況を再現する

### 3. 活用メリット（約2分）

- **2〜3つのメリット**を提示（3つに固執しない）
- 各メリットは**具体的な使用シーンと効果**をセットで説明
  - NG: 「情報検索が速くなります」
  - OK: 「『この型番の単価は？』と質問するだけで、AIが一瞬で答えてくれます。お客様をお待たせすることもなく、スムーズな対応が可能になります」
- **汎用性**は具体的な応用例を挙げて説明
  - 例: 「建材カタログだけでなく、設備のマニュアルや仕様書、社内の資料など、あらゆる文書に応用できます」

### 4. 本題への導入（約30秒）

- 使用する**ツールの機能を具体的に説明**
  - 例: 「PDFなどの資料を読み込んで、その内容について質問できるという便利な機能があります」
- **使用上の注意・制限事項を調査して明記**
  - 容量制限、データ制限、無料版/有料版の違いなど
  - 注意事項は本文内または※印の注釈として記載

### 5. 実演パートのアウトライン

- 大まかな手順の列挙のみ（詳細な台本は不要）
- **無料版と有料版**それぞれの使い方を提示
- 各手順は箇条書きで簡潔に

### 6. クロージング（約1分）

- ツールの価値を再確認
- **汎用性を具体例とともに**再度強調
  - 例: 「この方法は建材カタログだけでなく、設備のマニュアルや施工の手順書、社内の業務マニュアルなど、あらゆる資料に応用できます」
- 実務への応用を促す
- **終了文言**: 「【終わり】それでは次の講座で会いましょう。ありがとうございました。」

## 文体・トーンのルール

### 基本方針
- 完全な初心者向けを意識
- 専門用語は極力避け、使う場合は必ず平易な説明を付ける
- 親しみやすく、でも丁寧な口調

### 専門用語の説明パターン
| 専門用語 | 説明の付け方 |
|----------|-------------|
| Gemini | 「Gemini、つまりGoogleが提供している生成AIツール」 |
| NotebookLM | 「NotebookLM、つまりGoogleが提供しているAIツール」 |
| 壁打ち | 「壁打ち、つまり考えを整理するための相談相手として」 |
| プロンプト | 「指示文」と言い換える |
| GPTs | 「カスタム機能」と言い換える |
| 掛け率 | 「掛け率、つまり定価に対する実際の価格の割合」 |

### 住宅業界特有の表現例
- 「住宅会社の現場では、営業活動やお客様対応、現場管理など日々やるべきことが山積みで〜」
- 「特に、お客様との面談は非常に重要ですよね」
- 「お客様一人ひとりに合わせた丁寧な対応が求められます」

## AIツール情報の調査ルール

### 本題導入時の調査項目

台本作成時、使用するAIツールについて以下を調査し、注意事項として記載する:

1. **データ容量の制限**（アップロード可能なファイルサイズ上限）
2. **文字数・単語数の制限**
3. **ソース数の制限**（読み込める資料の数）
4. **無料版と有料版の機能差**
5. **その他の利用上の注意点**

### 記載形式

```
注意：1つのソースには最大〇〇語を含めることができます。
アップロードできるファイルのサイズは〇〇MBまでです。
最大〇〇個のソースを含めることができます。
```

## 品質チェックリスト

### 共通チェック項目

- [ ] 開始文言が「【スタート】こんにちは、Build AI Academyの〇〇です」になっている
- [ ] 終了文言が「【終わり】それでは次の講座で会いましょう。ありがとうございました。」になっている
- [ ] 専門用語・横文字にすべて説明が付いている
- [ ] 全体が初心者にも理解しやすい表現になっている
- [ ] 住宅会社目線の内容が含まれている

### ユースケース系追加チェック項目

- [ ] テーマがキャッチーなフレーズで提示されている
- [ ] 問いかけがリアルな場面描写になっている
- [ ] 課題の本質が明示されている
- [ ] 活用メリットが2〜3つで、具体的なシーンとともに説明されている
- [ ] 汎用性が具体的な応用例とともに説明されている
- [ ] ツールの使用上の制限・注意事項が記載されている
- [ ] 実演パートは大まかなアウトラインのみになっている
- [ ] 無料版/有料版の両方が提示されている

## 作業フロー

1. ユーザーからドラフトを受け取る
2. 台本の種類（ユースケース系/概念系）を判別する
3. 該当する構成に沿って内容を整理する
4. AIツールを使用する場合は制限事項を調査する
5. 台本として仕上げる
6. 品質チェックリストで確認する
7. 種類に応じたフォルダにMDファイルとして保存する
   - ユースケース系 → `作成済台本/ユースケース/`
   - 概念系 → `作成済台本/概念系/`

## フィードバック処理

### フィードバック受領時の対応手順

| ステップ | 内容 |
|----------|------|
| 1 | フィードバック内容を確認し、分類する（構成/文体/内容/フォーマット） |
| 2 | CLAUDE.mdの該当セクションを特定する |
| 3 | Editツールを使用してCLAUDE.mdを更新する |
| 4 | ユーザーに更新内容を報告する |
| 5 | 以降の作業で新ルールを適用する |
